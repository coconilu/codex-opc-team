# Contributing

感谢你参与 Codex OPC Team。贡献应守住三个核心边界：Codex 仍是 Harness、默认模式不依赖 Mem0、私人知识不得进入公开仓库。

## 开始之前

1. 阅读 [愿景与范围](docs/vision-and-scope.md)、[架构](docs/architecture.md)和相关 [ADR](docs/adr/README.md)。
2. 在 Issue 中说明用户问题、范围、验收条件和可能涉及的数据边界。
3. 对架构、数据模型、全局配置或安全边界的变更，先提交新的 ADR 或更新现有 ADR。

## 变更要求

| 变更类型 | 最低证据 |
|---|---|
| Skill 或工作流 | 成功路径与失败路径测试；不得依赖实现者自述 |
| Hook | 非 OPC 项目不记录事件的隔离测试；日志字段检查 |
| 记忆后端 | 无 Mem0、禁用 Mem0、后端故障降级测试 |
| 安装/卸载 | 重复执行安全；卸载后私人知识保留 |
| 全局配置 | 默认不修改；若用户明确选择，必须支持预览、备份和恢复 |
| 文档 | 命令、路径和能力状态与当前实现一致 |

## 本地工作流

```text
建立小范围变更
  → 添加或更新测试
  → 运行隐私扫描和验证
  → 更新文档/CHANGELOG
  → 提交 Pull Request
  → 独立评审与 QA
```

请使用小而清晰的提交，不要把格式化、重构和行为变更混在一个提交里。Pull Request 至少应包含：

- 问题和预期结果；
- 实际变更范围；
- 执行过的验证及结果；
- 风险和回滚方式；
- 是否处理用户数据、Hook 数据或 Codex 配置。

## 文档语言

英文 `README.md` 是公共主入口，`README.zh-CN.md` 提供完整中文入口。核心设计文档当前以中文维护；新增用户可见能力时应同步更新两个 README。

## 隐私红线

提交前确认变更中没有：凭据、真实聊天、经理档案、项目私有知识、用户名目录、本机绝对路径、session/turn ID 或未脱敏日志。发现安全问题时遵循 [SECURITY.md](SECURITY.md)，不要直接提交公开 Issue。
