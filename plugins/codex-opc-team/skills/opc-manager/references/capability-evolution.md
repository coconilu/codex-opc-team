# Capability evolution handoff

When a validated lesson proposes changing a role, Skill, or organization policy, keep the normal run and the capability proposal separate. Read the exact proposal/record through `opc_evolution.py`; do not infer active behavior from a candidate commit or an unstaged working tree.

Manager authority is required twice: first for the exact bounded pilot, then for the exact promotion target and version after conclusive beneficial evidence and independent QA. Promotion/rollback commands never stage or commit. Present the one returned pathspec and diff, require an explicit Git commit, then confirm its exact HEAD. Any other dirty path, missing evidence, unavailable Provider, timeout, scope/privacy failure, regression, or inconclusive result stops promotion.

After promotion, schedule the declared observation count. A rollback restores the previous capability bytes but retains pilot/evaluation/history and does not rewrite approved knowledge. Report only `association/evidence only`.
