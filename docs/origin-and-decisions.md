# 缘起与决策

## 1. 问题不是“再做一个 Agent”

这个项目源于一种长期使用 Codex 后的落差：每个新项目都需要用户从头到尾推动 vibe coding。模型能完成具体任务，却没有形成一个能长期理解经理意图、内部协作、独立验收并积累经验的团队。

目标体验不是“更长的自动化脚本”，而是：

```mermaid
flowchart LR
    I["经理表达想法"] --> A["团队感知并梳理"]
    A --> C["与经理对齐关键方向"]
    C --> W["内部委派和执行"]
    W --> Q["团队自行验收"]
    Q --> E["通过后邀请经理体验"]
    E --> L["沉淀可复用经验"]
```

其中“有生命力”有可验证的含义：团队能保留项目连续性，能在新任务中召回经过批准的经验，能根据真实结果修正做法，同时不会把偶然成功或错误结论自动固化为组织规则。

## 2. 两条最初思路

| 思路 | 核心隐喻 | 优点 | 风险 |
|---|---|---|---|
| 外包市场 | 用户拥有一个带记忆和反馈的招聘市场，每个项目招聘产品、开发、测试、运营等角色 | 角色灵活，贴近真实招聘和项目制协作 | 容易把重点放在角色数量，项目之间的治理和经验连续性不足 |
| 总管项目 | 一个长期存在的企业级总管项目维护配置、记忆和反馈，并管理其他项目 | 组织连续性强，便于治理和统一验收 | 容易发展成另一个独立平台，和 Codex 重复建设 Harness |

最终设计吸收两者的长处：

- 总管负责跨项目运行机制、对齐、委派和治理；
- 角色按项目和任务生成，不追求永久运行的 Agent 进程；
- 组织知识独立于插件代码和单个项目；
- Codex 继续负责执行环境、工具和子 Agent 编排。

## 3. 为什么不继续另起炉灶

已有独立 Agent MVP 的探索证明：重新实现一个运行时会把精力消耗在浏览器、文件系统、联网、工具调用、权限和会话管理上。这些正是 Codex 已经提供且用户依赖的能力。

因此系统边界调整为：

| Codex 负责 | OPC 插件负责 |
|---|---|
| 模型调用、文件和命令执行 | 经理—总管—角色—QA 的工作协议 |
| 浏览器、网络和工具 | 项目启动、验收契约和交接结构 |
| 子 Agent 生命周期 | 角色职责、委派与升级规则 |
| Codex 自身安全边界 | 组织记忆、反馈治理与召回策略 |

这一选择减少重复工程，也让插件能持续获得 Codex Harness 的能力演进。

## 4. 从本地原型到开源项目

第一版原型通过修改个人 Codex 配置、建立本地 `opc-knowledge`、安装个人 Marketplace 插件来验证工作流。它证明了经理、开发、独立 QA 和复盘链路可以运行，但也暴露了不可分发的问题：

- 角色配置包含本机绝对路径；
- 插件已经直接安装到个人环境，缺少可重复安装流程；
- 公开代码和私人知识边界尚未产品化；
- Hook 在确认 OPC 运行范围前记录事件，存在越界采集风险；
- Memory 主要依靠文件约定，还缺少统一接口和可选召回后端。

开源重构不是把原目录整体上传，而是选择性迁移通用行为、Schema 和脱敏模板，重新建立安装、隐私、测试和发布契约。

## 6. 贯穿项目的四个关键词

| 关键词 | 在 OPC 中的具体落点 |
|---|---|
| 上下文 | 基于目标、项目状态、验收契约和相关经验构造最小充分 Context Packet |
| 记忆 | 事实、经历和流程知识以可审计文件持久化；可选建立语义索引 |
| 反馈 | 构建、测试、浏览器验证、经理体验和失败证据进入闭环 |
| 进化 | 只有通过验证与批准的经验才改变未来行为，并且可追踪、可回滚 |

这四层不是相互替代：没有高质量上下文，记忆无法正确使用；没有反馈，记忆只会累积；没有治理，“进化”会退化为不可控的提示词漂移。
