# OPC Dashboard

OPC Dashboard 是一个显式启动、本机访问、只读的经理视图。它把当前项目契约、活动运行、验收、知识治理、经理待办、知识链路和 Provider 健康度投影到一个页面，目标是减少在多个 JSON、Markdown 与命令输出之间切换的成本。侧栏会切换独立视图，URL 锚点会保留当前视图，刷新或浏览器前进、后退后仍能回到原位置。

它不是稳定版 `v0.1.0` 的能力，也不是常驻服务。当前实现位于 `main`，必须由用户在需要时启动，关闭进程后即停止。

## 1. 最短体验路径

从仓库根目录运行：

```powershell
python plugins/codex-opc-team/scripts/opc_dashboard.py --project-root .
```

命令默认只绑定 `127.0.0.1:8569`，打印本地 URL，并尝试用默认浏览器打开。未自动打开时手动访问 `http://127.0.0.1:8569/`；按 `Ctrl+C` 停止。自动化或只想复制 URL 时可加 `--no-open`。如端口确实发生冲突，可显式使用 `--port` 临时覆盖。

如果只想确认布局和交互，不读取当前项目：

```powershell
python plugins/codex-opc-team/scripts/opc_dashboard.py --demo
```

## 2. 页面信息

| 区域 | 回答的问题 | 用户下一步 |
|---|---|---|
| 概览卡片 | 有多少活动项目、待验收项、知识候选和已发布知识 | 快速判断是否需要介入 |
| 项目状态 | 当前项目进行到哪一步，验收与反馈状态如何 | 回到对应项目处理 |
| 当前运行 | 每个显式接入项目当前运行的状态、活动性和验收进度 | 识别停滞或待体验的运行 |
| 知识漏斗 | 候选、待提交批准、已发布、拒绝与失效各有多少 | 优先处理不能发布的批准项 |
| 经理队列 | 哪些事项需要决定，为什么需要决定 | 查看下一步说明，不在 Dashboard 内写入 |
| 知识链路 | 当前任务与知识、QA、反馈和结果的关联证据 | 识别证据缺口；不把关联解释为因果 |
| 系统健康 | File/Git、Mem0 与项目数据是否可用 | 只对可操作的异常显示警告 |

## 3. 数据与状态语义

Dashboard 只读取显式传入的项目根目录，以及现有 OPC 组件已经允许读取的私有数据根。返回浏览器前会转换成固定字段的展示 DTO；原始正文、绝对路径、会话标识、凭据、Hook 载荷和运行日志不会进入响应。

| 状态 | 含义 |
|---|---|
| Candidate | 尚未经过经理批准的知识候选 |
| Approved, uncommitted | 已批准但未绑定当前 File/Git HEAD，不能作为 Published |
| Published | 已批准，且来源 Commit 与当前权威仓库 HEAD 一致 |
| Association only | 链路只证明记录之间存在可验证关联，不证明知识造成了结果 |
| Mem0 disabled | 正常的可选配置，不算错误；查询继续以 File/Git 为准 |

当前项目运行注册表只保存活动运行，因此 Dashboard 不声称提供完整历史项目清单。历史 sidecar 仅作为证据读取，不能被伪装成一等项目记录。

## 4. 安全边界

| 边界 | 实现 |
|---|---|
| 网络 | 仅允许 loopback，默认 `127.0.0.1`；拒绝非本机 Host 与跨源 Origin |
| 方法 | 仅允许 `GET` 和 `HEAD` |
| 路由 | 固定静态资源和 `/api/snapshot` 白名单 |
| 浏览器 | 无 CORS、无远程资源、CSP、`nosniff`、禁止嵌入与缓存 |
| 数据 | 固定字段投影；不返回正文、绝对路径、秘密或原始记录 |
| 行为 | 没有批准、晋升、编辑、删除、提交或部署操作 |

如果未来需要写操作、远程访问、多用户权限或完整历史索引，应分别设计和验收，不在当前只读 MVP 上叠加。

## 5. 已知限制

- 页面按 30 秒间隔在可见时刷新；它不是实时事件流。
- 一个进程可读取多个显式项目根，但不会扫描整台机器自动发现项目。
- 当前没有持久化的历史运行注册表，因此“活动项目”只代表本次显式输入中可确认的活动运行。
- 可选数据源不存在时按降级状态展示，不制造红色告警。
