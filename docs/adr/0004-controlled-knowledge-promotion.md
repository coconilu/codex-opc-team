# ADR-0004：经验通过受控流程晋升

- **Status:** Accepted
- **Date:** 2026-07-13

## Context

“团队成长”需要从已完成或失败的工作中学习。但一次成功可能依赖偶然环境，一次用户纠正也可能只适用于当前项目。如果 Agent 自动把复盘摘要写成全局规则，长期知识会被噪声、错误甚至提示注入污染，而且难以解释行为为什么改变。

## Decision

复盘只能创建 `Candidate`。候选必须检查来源、适用范围、敏感信息、与现有知识的冲突，并通过重放、Shadow Evidence、重复观察或明确人工批准等验证方式，才能迁移为 `approved` 持久状态。

`approved` 不等于已发布。`Published` 是可验证的运行条件，不是第五个持久状态：只有当前知识仓库 Git HEAD 能从该条目的 canonical 路径读到相同内容哈希时，它才是 HEAD-verifiable Published knowledge。正常 File 召回、上下文导出和 Mem0 候选回读都只能使用这类已发布且仍有效的条目。候选、未提交的 approved、已拒绝和 `obsolete` 条目不会作为当前规则使用。每次晋升、替代和失效都通过 Git 记录，可审计和回滚。

自动化可以整理、去重和提出建议，但不能静默批准改变组织行为的规则。

## Consequences

### Positive

- 降低记忆污染和策略漂移；
- 行为变化有来源、证据和责任边界；
- 冲突不会被静默覆盖；
- 错误知识可以失效和回滚。

### Negative

- 成长速度低于全自动写入；
- 需要策展状态机和审核工作；
- 需要设计合理的批量审核，避免经理注意力负担。

## Rejected alternatives

- **每次复盘自动写全局规则：** 快但不可控，错误会跨项目放大。
- **只保存完整聊天供模型自己判断：** 噪声大、隐私风险高、缺少明确状态。
- **只靠点赞/点踩晋升：** 反馈信息不足以证明适用范围和可重复性。
- **永不形成程序记忆：** 无法实现跨项目成长，只能重复解决问题。
