# ADR-0006：独立 QA 后才向经理交接

- **Status:** Accepted
- **Date:** 2026-07-13

## Context

用户希望像经理一样把握方向，而不是在每次实现后承担第一轮测试。如果实现者在完成代码后自行宣布完成，常见结果是构建未验证、验收条件遗漏或只测试了自己预期的路径。多 Agent 只有产生职责独立和真实证据才有价值，否则只是增加对话。

## Decision

有实质实现变更的工作必须由独立于实现者的 QA 职责按 Acceptance Contract 验证真实产物。开发者自测是输入证据，但不能给出最终 PASS。

QA 可以返回 PASS、FAIL 或 BLOCKED：PASS 后总管才向经理提供体验入口；FAIL 必须给出可复现证据并回到实现循环；BLOCKED 必须披露外部限制，不能伪装成 PASS。QA 不能自行降低或改写经理批准的验收标准。

对于纯解释、只读状态报告或无可验证实现的请求，总管可以按风险比例缩小 QA，但必须明确证据基础。

## Consequences

### Positive

- 经理接收到的是已验证结果而非实现者承诺；
- 缺陷更早暴露，失败有清晰回路；
- 角色分离带来实际质量价值；
- 验收标准成为团队共同契约。

### Negative

- 需要额外 Agent 时间和计算成本；
- 环境不足时可能产生 BLOCKED；
- 小变更需要按风险校准，避免流程过重。

## Rejected alternatives

- **实现者自测即完成：** 缺少独立视角，经理仍是实际 QA。
- **经理始终承担最终测试前的第一轮验证：** 与经理优先目标冲突。
- **QA 根据开发报告打分：** 没有验证真实产物，不构成验收。
- **所有任务固定同样重量的 QA：** 对只读或低风险任务成本不成比例。
