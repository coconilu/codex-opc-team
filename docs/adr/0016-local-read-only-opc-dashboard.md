# ADR 0016：本地只读 OPC Dashboard

- 状态：Accepted
- 日期：2026-07-23

## 背景

经理需要在项目契约、活动运行、验收文件、知识治理状态、反馈、知识链路与 Doctor 输出之间切换。原始文件和 CLI 对工程排障有效，但不适合快速判断“现在发生了什么、哪里需要我决定”。

Dashboard 同时带来新的隐私与产品风险：把私人正文或本机路径送进浏览器、把可选 Mem0 误报为故障、把证据关联说成因果，以及为了一个只读视图引入前端构建链、数据库或常驻服务。

## 决策

采用 Python 标准库本地服务和无构建步骤的单页静态界面：

1. 只允许 loopback，默认和文档化入口使用 `127.0.0.1`；用户显式启动和停止。
2. 只接受显式 `--project-root`；不扫描磁盘发现项目。
3. File/Git 仍是知识权威源，Mem0 只是可选、可重建的 Provider。
4. 复用现有治理、Doctor、反馈和链路读取逻辑；浏览器只接收固定字段的只读 DTO。
5. 固定路由、`GET`/`HEAD` 方法和安全响应头；不启用 CORS、远程资源、遥测或响应缓存。
6. 页面不提供批准、晋升、编辑、提交、删除或部署操作。
7. 链路始终标记 `association/evidence only`，不得推断采用或因果。
8. 当前运行存储只支持活动运行时，页面明确降级，不声称拥有完整历史。

## 结果

正面结果：

- 用户用一个页面完成日常态势判断，仍可回到现有 CLI 执行受控操作。
- 零 Node、零数据库、零图表库依赖，安装与维护成本低。
- 私人数据先在服务端投影和脱敏，不把原始记录交给前端筛选。
- 缺失的可选数据源安全降级，不阻塞 File/Git 基线。

代价：

- 页面只在进程运行期间可用，不支持远程或多用户访问。
- 没有历史运行注册表时，不能展示完整项目历史。
- 写操作、身份鉴别、实时推送和高级分析需要独立 ADR 与验收。

## 未采用方案

| 方案 | 原因 |
|---|---|
| React/Next.js 与 Node 构建链 | 当前交互和状态规模不需要，引入额外安装与供应链成本 |
| SQLite 或新索引数据库 | 会产生第二事实来源和迁移问题 |
| WebSocket/常驻后台服务 | 当前 30 秒可见时刷新已满足经理态势判断 |
| 自动扫描用户目录 | 扩大隐私范围并使项目边界不可预测 |
| 在 Dashboard 内执行治理写操作 | 混淆只读理解与高风险决策，破坏现有确认门禁 |
